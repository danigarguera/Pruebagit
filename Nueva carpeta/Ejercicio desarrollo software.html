Cascade model

It is a sequential model where each phase of development (e.g., requirement analysis, system design, implementation, testing, deployment, and maintenance) follows the previous one. In this model the developers don't go back to a previous phase once completed, unless you start from scratch.

Some of the main characteristics of this model are:

  • Rigid and linear.
  • Requirements are fixed early.
  • Best for well-defined projects where requirements are unlikely to change.

Iterative model

This model focuses on repetition. The development starts with a simple initial version, and more features are added in iterations. Each iteration results in a new version of the product that includes enhancements and refinements.

The main characteristics of this model are:

  • Early iterations can reveal issues and provide early testing opportunities.
  • Allows for improvements over multiple cycles.

Incremental development model

This model is similar to the iterative model but involves building the software incrementally. The system is developed in smaller, manageable pieces (increments), with each increment delivering part of the functionality.

The key characteristics of the incremental model are:

  • The system is developed piece by piece, with each increment adding functionality.
  • Combines elements of linear and iterative models.

Spiral model

The spiral model combines elements of iterative development and systematic risk analysis. Each phase in the development process is preceded by a risk analysis step, ensuring that potential risks are identified and addressed early.

Some of the main characteristics of the spiral model are:

  • The development is broken down into multiple phases, with risk analysis being central to each phase.
  • It allows for changes and revisions based on risk findings.

Prototype model

The prototype model Involves building a prototype or working model of the software early in the development cycle to gather feedback from users. This prototype is typically a simplified version that helps clarify requirements.

Their main characteristics are:

  • Focuses on creating early versions of the software (prototypes) to visualize the final product.
  • The prototypes may be discarded after requirements are finalized, but feedback guides the final system.
